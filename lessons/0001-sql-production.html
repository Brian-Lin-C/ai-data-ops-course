1. 窗口函数 — 面试必杀技

学术 SQL 用 GROUP BY 聚合就够了，但生产环境中你需要保留每一行明细的同时做计算——这就是窗口函数。

四大金刚

函数作用典型场景
ROW_NUMBER()每行唯一序号（1,2,3...）去重取最新一条
RANK()排名，并列跳号（1,2,2,4...）销售额排名
DENSE_RANK()排名，并列不跳号（1,2,2,3...）连续排名
LAG() / LEAD()取前一行 / 后一行环比计算、连续判断

实战例题 ①：每个用户最近一笔订单

-- 面试高频题：取每个用户的最新订单
SELECT user_id, order_id, amount, order_time
FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY order_time DESC) AS rn
  FROM orders
) t
WHERE rn = 1;

实战例题 ②：计算次日留存率

-- 数据分析师必会：次日留存
WITH first_day AS (
  SELECT user_id, MIN(DATE(login_time)) AS first_date
  FROM user_login
  GROUP BY user_id
)
SELECT 
  f.first_date,
  COUNT(DISTINCT f.user_id) AS new_users,
  COUNT(DISTINCT l.user_id) AS day1_retained,
  ROUND(COUNT(DISTINCT l.user_id) * 100.0 / COUNT(DISTINCT f.user_id), 2) AS retention_pct
FROM first_day f
LEFT JOIN user_login l 
  ON f.user_id = l.user_id 
  AND DATE(l.login_time) = DATE_ADD(f.first_date, INTERVAL 1 DAY)
GROUP BY f.first_date
ORDER BY f.first_date;

2. 多表 JOIN — 数据血缘的核心

生产环境的数据分散在几十张表里，JOIN 是连接它们的桥梁。

JOIN 类型结果使用场景
INNER JOIN两表都有匹配才保留核心业务关联（订单×用户）
LEFT JOIN左表全部保留，右表无匹配为 NULL主表兜底（所有用户×订单）
RIGHT JOIN右表全部保留较少用，可用 LEFT 替代
FULL OUTER JOIN两边都保留对账、全量对比
SELF JOIN表自己 JOIN 自己层级关系、连续判断

面试易错点：LEFT JOIN 后 WHERE 条件放哪里？

-- ❌ 错误：WHERE 过滤了 LEFT JOIN 的 NULL 行，退化成 INNER JOIN
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.amount > 100;  -- NULL 被过滤掉了！

-- ✅ 正确：条件放在 ON 里
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.amount > 100;
-- 保留了没有订单的用户（amount 为 NULL）

3. CTE（WITH 子句） — 可读性的分水岭

见过嵌套 4 层的子查询吗？CTE 就是解药。学术 SQL 很少用，但生产代码里到处都是。

-- ❌ 难以阅读的嵌套子查询
SELECT ...
FROM (
  SELECT ...
  FROM (
    SELECT ...
    FROM orders
  ) a
) b;

-- ✅ 用 CTE 改写 —— 清晰可读
WITH 
  user_orders AS (
    SELECT user_id, COUNT(*) AS order_cnt
    FROM orders
    GROUP BY user_id
  ),
  high_value AS (
    SELECT * FROM user_orders WHERE order_cnt >= 5
  )
SELECT u.name, h.order_cnt
FROM users u
JOIN high_value h ON u.id = h.user_id;

4. Hive SQL — 互联网公司的标配

JD 上写「SQL」其实 70% 是 Hive SQL。互联网公司的数据在 Hadoop/Hive 上，不是 MySQL。

核心差异速查

分区表PARTITIONED BY (dt STRING)，查询必须带 WHERE dt = '2025-06-01' 否则全表扫描
collect_list聚合为数组：collect_list(product_name)，MySQL 没有
lateral view explode炸开数组：LATERAL VIEW EXPLODE(tags) t AS tag
等值 JOIN 限制只支持 = 的 JOIN 条件，不支持 > < <>（需要放 WHERE）
INSERT 不同Hive 是批量追加，不是逐行插入
-- Hive 典型写法：分区查询 + collect_list
SELECT 
  user_id,
  collect_list(order_id) AS order_ids,
  COUNT(*) AS order_cnt
FROM dwd_order_info
WHERE dt = '2025-06-22'  -- 必须带分区！
GROUP BY user_id;
